# ADR 0013: ASIO Identity-Bound Report Access

## Status

Accepted direction from discussion.

## Context

`main` added a shared tester access-code gate using:

- `APP_ACCESS_TOKEN_HASH`;
- `APP_ACCESS_CODE_LABEL`;
- constant-time hash comparison;
- hosted session start before chat;
- smoke coverage for a rejected wrong access code.

That model fits a standalone hosted web MVP. It does not fit the current product direction.

The accepted interface direction in ADR 0012 is that users enter through ASI/ASI:ONE. The FieldBrief Agent in the frontend is only a development/debug simulation of that entry. After ASI/ASI:ONE produces a report, the user may need to open a detailed report view through a link. That report view must be protected by an identity and authorization boundary strongly tied to the ASI/ASI:ONE user/session/case context.

A shared demo access code is not enough for detailed report access. It is not user-specific, not case-specific, and not appropriate as a production or product identity boundary.

This ADR is parallel to ADR 0012, ADR 0014, ADR 0015, and ADR 0016. It decides only the report access and identity boundary. It does not decide upload storage, session persistence, or deployment mechanics.

## Decision

Do not implement Evan-style shared access-code session start.

Use an ASI/ASI:ONE identity-bound access model for detailed report viewing. The report link generated after a confirmed ASI/ASI:ONE workflow should not be treated as a bearer secret by itself. Opening the report must require an authenticated ASI/ASI:ONE identity context, or an ASI-issued short-lived access context that can be validated before report lookup.

The report access check should happen before returning stored report details from AgentCore. It should validate that the requester is allowed to view the requested `caseId`.

The validation model should include, at minimum:

- authenticated ASI/ASI:ONE subject or session identity;
- requested `caseId`;
- binding between that identity/session and the case/report generated by the entry-agent workflow;
- expiry or revocation semantics for any short-lived report-view token;
- denial before `rams_supervisor_runtime` or report storage returns report details.

The FieldBrief Agent development/debug entry may bypass this production identity model only in explicit local development mode. That bypass must be documented as dev-only and must not define product behavior.

## Options Considered

1. Restore Evan's shared access-code gate.
   - Pros: easy to demo and already understood.
   - Cons: not ASI identity-bound, not case-specific, and inappropriate for report detail access.

2. Treat `/case/{caseId}` as public if the id is hard to guess.
   - Pros: simplest link model.
   - Cons: turns `caseId` into a bearer secret and weakens privacy, audit, and authorization boundaries.

3. Use ASI/ASI:ONE identity-bound report access.
   - Pros: aligns with the actual product entry path and makes report detail access user/session/case aware.
   - Cons: requires an ASI identity assertion, token, or callback contract that is outside the old FastAPI MVP.

## Consequences

Positive:

- The report detail view is protected by the same product identity context that launched the workflow.
- `caseId` remains a correlation id, not a secret access token.
- FieldBrief remains a development/debug tool instead of becoming a parallel product login flow.
- AgentCore report lookup can enforce authorization before returning detailed output.

Tradeoffs:

- Local debugging needs an explicit bypass or fixture identity.
- Hosted smoke tests must simulate or provide a valid ASI/ASI:ONE identity context.
- The exact ASI identity assertion format must be designed or integrated when ASI/ASI:ONE access is available.

## Acceptance Criteria

- The codebase does not reintroduce `/api/session/start` or shared-code login as the product access model.
- Report links include or resolve to `caseId`, but `caseId` alone is not sufficient to retrieve detailed report content in hosted mode.
- Report lookup validates ASI/ASI:ONE identity/session/case authorization before returning stored `run` or `structuredReport`.
- Dev-only FieldBrief report lookup is clearly gated behind local/debug configuration.
- No raw ASI identity tokens, access assertions, private user data, or report-view credentials are written into public docs, traces, or logs.
- Public docs state that report access control is ASI/ASI:ONE-bound and that FieldBrief is a development/debug simulation only.

## Discussion Questions

- What concrete ASI/ASI:ONE identity artifact will the report viewer receive: signed JWT, short-lived report token, callback session, or another assertion?
- Should the report viewer be hosted inside ASI/ASI:ONE, or can it be a separate frontend that validates ASI-issued access before calling AgentCore?
- Should authorization be checked in the signed proxy, in `asi_one_entry_agent`, or in a dedicated report-access adapter before invoking supervisor report lookup?
